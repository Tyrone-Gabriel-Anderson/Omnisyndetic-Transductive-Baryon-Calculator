# The Omnisyndetic Framework Developer Project

This repository contains the Unity-based developer environment for the wider **Omnisyndetic Framework** project.

At its centre is a **pre-dynamical geometric baryon calculator**. The tool is intended to give a direct, runnable way of exploring the baryonic side of the project through geometry rather than through a purely textual account. It is public so that anyone can download it, inspect it, test it, and use it as a practical entry point into the wider work.

The broader Omnisyndetic project explores whether some aspects of baryonic organisation and associated mass structure can be approached from a fixed relational and geometric construction, rather than beginning with intrinsic constituent masses, fitted breaking terms, or state-by-state adjustment. The present repository should be read in that exploratory spirit. It is a working environment for learning, testing, auditing, and further development.

## What this repository is for

This repository is here as a public Unity-side workspace for the Omnisyndetic project.

It is intended for:

- **learning**, by giving a direct way to explore the structure through a runnable tool,
- **testing**, by allowing users to enter states and inspect the resulting outputs first-hand,
- **auditing**, by making the implementation open to inspection and comparison with the written materials,
- **development**, as the toolset and wider project continue to grow,
- **inspiration**, for anyone who may wish to build related visual, geometric, or structural tools of their own.

This is not meant to be a sealed showcase. It is a public working environment.

## What the tool delivers

The tool provides a **pre-dynamical geometric calculation of baryonic structure**.

A user can enter either:

- the **quark flavour content** of a baryon, or
- the equivalent **structural values** used by the framework.

From there, the calculator resolves the same state under the same fixed construction. The intention is that the route remains consistent across valid forms of input, without a separate hidden path behind the interface.

The tool then returns linked outputs showing:

- the geometric organisation associated with the chosen state,
- the allowed or excluded charge-tier behaviour visible from that organisation,
- the corresponding state-space displacement read-out,
- the triadic geometric view associated with the construction,
- and the resulting **rest-energy band** associated with that certified placement.

The result is given as a **band** rather than a single chosen number. The midpoint is shown only as the centre of the admitted interval.

So reading this plainly: the calculator lets a user enter flavour content, charge, and spin, then inspect how the fixed geometric construction resolves the placement and the corresponding baryonic mass interval.

## Pre-dynamical starting point

The derivation implemented here is **pre-dynamical** in a strict sense.

It does not begin by assuming space and time as primitive substrates of the construction, and it does not start from a prior spacetime background into which the baryon is placed. Instead, it begins from a fixed relational and geometric register, from which admissible organisation and mass-band assignment are read.

That starting point is part of the interest of the project. It keeps the present work at the level of prior structure, while leaving open the possibility that later work may examine how dynamical, relativistic, or symmetry-group descriptions relate to that same foundation.

This repository should therefore be read as a practical implementation of a pre-dynamical geometric route, not as a finished account of everything that might later grow from it.

## Geometric and relational numerics

A central idea of the project is that the numerics used by the baryon calculator are meant to come from the framework’s own fixed geometric and relational construction.

The interest of the tool lies partly here. It is not only that a band is produced, but that the band is intended to come from a geometric route internal to the framework itself.

In that sense, the calculator explores whether some of the structure often approached through fitted intrinsic masses, explicit breaking mechanisms, or Lie-group machinery might also be approached from a more prior geometric construction. That should be read carefully and modestly. The point here is not to dismiss standard approaches, but to provide a concrete environment in which a different route can be inspected and tested.

## What a user can do with it

### Reproduce known audited states

Users can enter established baryon flavour combinations and inspect the resulting geometric placement and associated rest-energy band under the fixed rules used by the calculator.

### Explore beyond the audited set

The same tool can also be used to test combinations beyond the presently audited census. This makes the repository useful not only for reproduction, but also for exploratory extension.

### Inspect the geometry directly

The Unity environment is intended to show more than a numerical output. It allows users to inspect how geometric placement, charge-tier restriction, state-space displacement, and band assignment sit together as one connected construction.

### Compare implementation and written material

Because the project is public, users can compare the executable tool against the wider written project and assess whether the implementation behaves in line with the stated structure.

### Build from it

The repository is also here for development. Others may wish to improve the interface, refine the presentation, extend the visualisation, experiment with related interaction models, or use the work as a starting point for nearby ideas of their own.

## Why this repository is public

A central aim of the wider project is that the work should be available for direct inspection.

Rather than asking people to engage only with summaries or descriptions, this repository lets them open the Unity project itself and explore how the geometric construction is being rendered in executable form.

That openness serves several purposes:

- it makes the project easier to learn,
- it allows the claims to be tested more directly,
- it supports transparent audit,
- and it creates room for further development by others.

That is the reason this repository is public.
